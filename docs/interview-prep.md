# 면접 활용 노트

이 문서는 KnitLog 프로젝트를 면접에서 설명하기 위한 노트다. 구현이 진행되는 동안 계속 업데이트한다.

## 1. 한 줄 소개
실제 뜨개 취미를 가진 사용자가 겪는 불편(도안/진행 기록 산재, 게이지 환산, 실·바늘 재고 관리)을 해결하기 위한 개인용 니팅 관리 백엔드. 단순 CRUD 게시판이 아니라 도메인 모델링, JPA 연관관계 설계, 비즈니스 로직, 인증/인가, 배포까지 아우르는 것을 목표로 한 포트폴리오 프로젝트.

## 2. 시작 배경 (왜 이 프로젝트인가)
- 실제로 뜨개질을 하며 겪은 문제에서 출발 — 임의로 정한 튜토리얼 주제가 아니라 실사용 목적이 있다는 점
- 해결하려는 문제: 도안/진행 기록이 여러 곳에 흩어짐, 게이지 기반 코수/단수 환산이 번거로움, 보유 실 재고 파악이 어려움, 바늘/케이블 재고를 별도로 관리하기 어려움
- 7.5년차 Java 웹 개발자(JSP/MyBatis/Oracle/jQuery/AngularJS 중심)로서, 모던 스택(JPA/JWT/Docker/클라우드)으로 이직을 준비하며 실무 경험이 없는 부분을 이 프로젝트로 채우는 중

## 4. 아키텍처 / 설계 결정 포인트 — 질문 대비

### 4.1 레이어드 아키텍처 (Controller-Service-Repository-DTO)
- 결정: Entity를 API 응답에 직접 노출하지 않고 DTO로 변환, Controller/Service/Repository 책임 분리
- 이유: Entity 변경이 API 스펙에 그대로 전파되는 것을 막고, 지연 로딩 프록시가 직렬화 과정에서 문제를 일으키는 것을 방지. 각 레이어의 책임(요청 처리/비즈니스 로직/영속성)을 명확히 분리
- 예상 질문: "Entity를 그대로 응답하지 않는 이유는?", "DTO 변환은 어디서 하는 게 맞는가?"

### 4.2 YarnColor → YarnInfo 단방향 연관관계
- 결정: YarnInfo에 `List<YarnColor>` 역참조를 두지 않고 YarnColor → YarnInfo 단방향 `@ManyToOne`만 설계
- 이유: 아직 YarnInfo 입장에서 하위 색상 목록을 순회할 필요가 없고, 양방향으로 만들면 컬렉션 동기화(추가/삭제 시 양쪽 다 관리)에 대한 부담만 늘어남. 필요해지면 `YarnColorRepository.findByYarnInfoId`로 조회 가능
- 예상 질문: "양방향 연관관계로 만들지 않은 이유는?", "나중에 컬렉션 조회가 필요해지면 어떻게 확장할 것인가?"

### 4.3 meterPerGram을 컬럼으로 저장하지 않음
- 결정: `lengthPer100g`만 저장, g당 길이는 필요 시 계산
- 이유: 파생 가능한 값을 저장하면 원본 값이 바뀔 때 정합성이 깨질 위험(업데이트 누락)이 생김 — 정규화 원칙에 따른 선택
- 예상 질문: "왜 계산 필드를 별도로 저장하지 않았는가?", "조회 성능이 중요해지면 어떻게 할 것인가?"(캐싱/역정규화 트레이드오프로 답변)

### 4.4 YarnStock을 이벤트/히스토리 자체로 설계 (YarnStock + History 분리 안 함)
- 결정: 재고를 하나의 최신 상태 레코드로 관리하지 않고, 입고/사용/조정 등 변동 하나하나를 레코드로 남기는 이벤트 로그 방식
- 이유: 재고 변화 이력 자체가 도메인상 의미 있는 정보(언제 얼마나 늘고 줄었는지) — 현재 재고량은 이벤트 합산으로 조회
- 예상 질문: "현재 재고량은 어떻게 계산하는가?", "이벤트가 많아지면 성능은 어떻게 관리할 것인가?"

### 4.5 Wish → KnittingProject 생성 후 독립적으로 관리(스냅샷)
- 결정: Wish 기반으로 KnittingProject를 생성한 뒤에는 Wish 데이터를 참조하지 않고 값을 복사
- 이유: 원작 정보(Wish)가 나중에 수정되어도 이미 시작한 프로젝트의 기록은 영향받으면 안 됨 — 이력 보존이 목적
- 예상 질문: "왜 FK 참조 대신 값을 복사했는가?", "그로 인한 데이터 중복은 어떻게 보는가?"

### 4.6 ddl-auto: update / open-in-view: false
- 결정: 현재 학습 단계라 `ddl-auto: update`로 스키마를 자동 반영, `open-in-view: false`로 OSIV(Open Session In View)를 끔
- 이유: update는 초기 개발 속도를 위한 임시 선택(추후 Flyway 마이그레이션으로 전환 예정). OSIV를 끄는 이유는 지연 로딩이 Controller/View 레이어까지 열려있게 되는 것을 막아 트랜잭션 경계를 Service 레이어로 명확히 하기 위함
- 예상 질문: "OSIV를 끈 이유와 그로 인해 생기는 제약은?", "운영 환경에서는 ddl-auto를 어떻게 가져갈 것인가?"
